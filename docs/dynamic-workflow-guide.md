# Dynamic Workflow：由浅入深技术指南

> 本文解释 Dynamic Workflow 这一范式的技术逻辑——它解决什么问题、如何运转、以及每个设计选择背后的权衡。
> 不假设读者熟悉任何特定框架。

---

## 第一层：直觉篇 —— 为什么要这样做？

### 从一个厨师说起

想象一个厨房。一个厨师接到一桌 20 道菜的订单。

**方式 A：一个人做。** 他做第一道，再做第二道，再做第三道……做到第十道的时候，他已经忘了第一道的酱汁配方。这就是传统 LLM 对话的模式——所有中间结果堆在上下文里，越堆越多，最终溢出或变模糊。

**方式 B：写一份食谱，分给多个厨师。** 一个主厨写好食谱，每道菜配一份详细的操作步骤。多个厨师同时开工，各自独立完成。主厨不需要记住每道菜的中间状态——食谱记住了。这就是 Dynamic Workflow。

核心矛盾很简单：**当任务大到"一个人脑子装不下所有中间状态"时，你需要的不是更大的脑子，而是一份食谱。**

### 那为什么不用清单？

你可能会说：我写一个 YAML 清单，列出所有步骤，让 LLM 逐个执行不就行了？

清单对于线性流水线够用。但真实任务不是线性的：

- "审计所有 API 端点的安全性" —— 每个端点独立分析，但分析结果之间需要交叉验证
- "把这个项目从 REST 迁移到 GraphQL" —— 先调研现状，再分批改造，每批改造后要跑测试，失败的要重试
- "调查 Node.js v20 到 v22 的权限模型变化" —— 从多个角度搜索，找到的来源要互相印证，不可靠的要淘汰

这些任务有循环、有分支、有扇出扇入、有条件重试。清单是静态的，**你需要的是一段程序**。

### 从"谁持有计划"看四层演化

Claude Code 官方有一个很精准的观察：区分不同 agent 协作方式的本质问题是——**谁持有计划？**

| 机制 | 谁决定下一步 | 中间结果在哪 | 能做什么 | 规模 |
|------|------------|------------|---------|------|
| **Subagent** | LLM，每一轮临时决定 | LLM 上下文 | 分担单项任务 | 几个 |
| **Skill** | LLM，按预设指令 | LLM 上下文 | 可复用的专项能力 | 几个 |
| **Agent Team** | Lead agent，逐轮分配 | 共享任务列表 | 多个长期协作者 | 几个 peer |
| **Workflow** | 脚本代码 | 脚本变量 | 可编程的编排逻辑 | 几十到几百 |

演化方向很清晰：**计划从 LLM 的"脑子"里，一步步外化到"代码"里。**

- Subagent：LLM 即兴发挥，适合作小任务
- Skill：把重复的指令固化下来，但编排仍是 LLM 实时决策
- Agent Team：有了团队分工，但协调仍是逐轮对话
- **Workflow：编排逻辑彻底外化为代码**，LLM 只在每个 agent 调用时充当执行者

这个外化带来三个根本性的变化：

1. **规模突破**：中间结果不再受上下文窗口限制，脚本变量没有容量上限
2. **确定性**：同样的脚本跑两次走同样的路径，可重跑、可恢复、可审计
3. **质量模式**：脚本可以内建"对抗式验证"这类需要多 agent 结构化协作的模式，这不是单次 LLM 调用能做到的

### 一句话总结

**Dynamic Workflow 把"谁持有计划"从模型下沉为代码，让 LLM 从"边想边做的决策者"变成"按指令执行的工人"，从而突破上下文限制、获得编排确定性、并支持结构化的质量保证模式。**

## 第二层：机制篇 —— 它是怎么运转的？

### 整体画面

```
用户描述一个任务
      │
      ▼
  LLM 生成编排脚本          ← LLM 唯一一次参与编排决策
      │
      ▼
  运行时启动，后台执行        ← 独立于用户会话
      │
      │  脚本执行到 agent() 调用
      ▼
  调度器派发任务给 LLM       ← LLM 此刻只是执行者
      │
      │  执行结果返回脚本变量
      ▼
  脚本继续下一步              ← 循环、分支、扇出……
      │
      │  全部完成
      ▼
  汇总报告返回用户
```

关键点：LLM 参与了两次——一次写脚本，一次（或多次）作为 agent 执行任务。但编排逻辑从头到尾在脚本里，不在 LLM 的上下文里。

### 五个核心机制

#### 机制一：上下文外化

传统模式下，一个 agent 的输出是另一下 agent 的输入，中间结果都经过 LLM 上下文：

```
LLM 上下文: [系统提示 | 任务描述 | Agent1结果 | Agent2结果 | Agent3结果 | ... ]
                                                              ↑ 越来越大，最终溢出
```

Workflow 模式下，中间结果存在脚本变量里：

```
脚本变量: result1 = {...}, result2 = {...}, result3 = {...}
LLM 上下文: [系统提示 | 当前这一个 agent 的任务描述]  ← 始终很小
```

每个 agent 看到的上下文只有自己的任务描述。它不知道其他 agent 的存在，也不需要知道。脚本变量充当了一个无限大的"外部记忆"。

#### 机制二：脚本沙箱

脚本**只能协调，不能执行**。它不能读文件、不能发网络请求、不能运行命令。

所有实际的 IO 都由 agent 完成。脚本的工作就是：

- 决定派什么任务给哪个 agent
- 收集结果
- 根据结果决定下一步
- 汇总最终报告

这就像一个项目经理——他写计划、分配任务、收集报告，但他不亲自写代码。实际干活的是 agent。

为什么这样设计？因为脚本是 LLM 生成的。如果你让 LLM 生成的脚本有文件系统访问权，就等于给了一个不可信程序完整的系统权限。沙箱把这个风险降到零。

#### 机制三：确定性编排

脚本是代码。代码的特点是：**同样的输入，同样的执行路径。**

这意味着：

- **可重跑**：脚本保存下来，下次同样的任务直接执行，不需要再让 LLM 重新规划
- **可恢复**：执行到一半崩溃了，重新启动时跳过已完成的 agent（通过缓存），只跑没完成的
- **可审计**：脚本是文本文件，可以 diff、review、版本控制
- **可调优**：结果不理想？修改脚本里的参数或逻辑，重新跑

对比 LLM 逐轮决策的不确定性——同一个任务跑两次可能走完全不同的路径——脚本的确定性是一个本质提升。

#### 机制四：质量模式

脚本不只是调度器。它可以编排**多个 agent 之间的结构化协作**，实现单次 LLM 调用做不到的质量保证。

最典型的是**对抗性收敛**：

```
第一轮：
  Producer 们各自独立分析 → 产出一批 Findings
  Adversary 们尝试反驳每个 Finding
  投票：被成功反驳的淘汰

第二轮：
  存活的 Finding 再次被新的 Adversary 攻击
  再次投票淘汰

… 直到没有新的被淘汰（收敛）或达到最大轮数
```

这本质上是**科学同行评审的自动化**：

- **多视角**：多个 producer 独立分析同一个问题，避免盲点
- **故意挑错**：adversary 的唯一任务就是找漏洞
- **投票共识**：只有经过多轮考验的结论才能存活

单次 LLM 调用可能有 70% 准确率。但经过 3 轮对抗收敛，存活下来的结论是高度可信的。

为什么这必须靠脚本来编排？因为这个过程有循环（多轮迭代）、有条件分支（是否收敛）、有 fan-out（多个 producer/adversary 并行）、有状态追踪（哪些 finding 存活）。这不是一个线性清单能描述的——你需要一段程序。

#### 机制五：后台执行与可观测性

Workflow 运行时独立于用户会话。用户启动一个 workflow 后，可以继续做别的事。

运行时通过**事件总线**广播状态：哪个 phase 开始了、哪个 agent 在跑、花了多少 token、哪个失败了。用户随时可以查看进度、暂停、恢复、甚至停止单个 agent。

这和传统模式（逐轮对话，用户必须等每一步完成才能继续）形成鲜明对比。

### 这五个机制的因果关系

```
上下文外化 → 规模突破（从几个到几百个）
    │
确定性编排 → 可重跑/可恢复/可审计
    │
质量模式 → 从"跑更多"到"跑出更可信的"
    │
后台执行 → 用户不被阻塞
    │
脚本沙箱 → LLM 生成的脚本不会造成安全风险
```

这五个机制不是独立的功能列表，而是一个连贯的设计：上下文外化是起点，确定性编排是基础，质量模式是核心价值，后台执行和沙箱分别是用户体验和安全保障。

## 第三层：深度篇 —— 关键设计为什么是这样？

这一层讨论每个设计决策背后的权衡——为什么选 A 而不是 B。

### 决策一：为什么用脚本语言而不是 YAML DSL？

很多编排系统用 YAML 或 JSON 描述工作流（GitHub Actions、Airflow DAG）。为什么 Dynamic Workflow 不这样做？

**YAML 能做的事：** 线性步骤、简单分支、扇出扇入

**YAML 做不到的事：**

- 动态决定"分析哪些文件"——你不知道代码库里有多少文件需要扫描，这是运行时才能确定的
- 根据上一轮结果决定下一轮的策略——"如果发现了 3 个以上的高危漏洞，进入深度审查模式"
- 循环直到收敛——对抗性收敛的轮数是运行时动态决定的

这些都需要**图灵完备的编程语言**。

但为什么是脚本语言（Lua/JS），而不是 Python 或 Go？

- **Python**：标准库太强大（文件系统、网络、子进程），沙箱化成本极高且不彻底
- **Go/Rust**：编译型语言，修改→编译→运行的循环太重
- **Lua/JS**：解释型、标准库小、沙箱化容易、热加载

选 Lua 还是 JS 是实现差异（Claude Code 选了 JS，Maestro 选了 Lua），但"用可沙箱化的脚本语言"这个决策是一致的。

### 决策二：为什么是 Prompt-in / Result-out 的单次模型？

Dynamic Workflow 里的每个 agent 调用都是**单次的**：给一个 prompt，返回一个 result。不是多轮对话。

为什么不做多轮交互？因为多轮交互和"确定性编排"矛盾。

考虑多轮模式：agent 在执行过程中遇到问题，需要追加信息。这时谁来回答？如果让 LLM（编排者）回答，编排逻辑又回到了 LLM 上下文里，你失去了确定性。如果让用户回答，你失去了后台执行的能力。

单次模型牺牲了 agent 的灵活性（它不能追问），换来的是：

- **可缓存**：相同的 prompt 产生相同的 cache key，重复执行直接跳过
- **可重试**：失败后幂等地重新提交整个任务，不需要恢复对话状态
- **可并行**：N 个独立任务同时派出，互不干扰
- **可恢复**：崩溃重启后，根据 cache key 精确跳过已完成的调用

这是一个经典的工程权衡：**限制单次交互的灵活性，换取系统级的可靠性。**

实践中，这个限制通过把任务描述写得足够详细来弥补——如果 agent 需要追问，说明你的 prompt 不够好。

### 决策三：对抗性收敛的代价曲线

对抗性收敛强大，但它有真实成本。

假设你要分析 10 个文件，每个文件 3 个 producer、2 个 adversary、最多 3 轮：

```
Producer 调用: 10 × 3 = 30 次
Adversary 调用: (假设每轮产生 5 个 finding) 5 × 2 × 3 = 30 次
总计: ~60 次 LLM 调用

对比: 简单 parallel 只需 10 次调用
成本倍数: 6×
```

什么时候值得付出这个代价？

| 场景 | 推荐模式 | 理由 |
|------|---------|------|
| 格式转换、批量重命名 | parallel | 结果是确定性的，不需要验证 |
| 代码审查 | converge（1 轮） | 多视角有价值，但一轮通常够 |
| 安全审计 | converge（3 轮） | 漏报的代价极高，值得多轮验证 |
| 竞品分析 | converge（2-3 轮） | 需要交叉验证多个来源的可靠性 |
| 调研报告 | deep-research 模式 | 搜索→交叉验证→综合，本质上就是 converge |

一个实用的经验法则：**如果错误的代价远高于多跑几个 agent 的成本，就用 converge。**

### 决策四：Cache Key 基于什么做哈希？

恢复（resume）依赖缓存。缓存的 key 决定了"什么算同一个调用"。

Cache key 由四个元素生成：

```
key = hash(backend_id + model + prompt + phase)
```

**为什么是这四个？**

- `backend_id`：不同的后端（Claude vs GPT-4）对同一 prompt 给不同结果
- `model`：同一后端的不同模型也是不同的
- `prompt`：核心输入，任何变化都意味着不同的调用
- `phase`：同一个 prompt 在不同阶段可能有不同的上下文含义

**为什么不是只有 prompt？** 因为同一个 prompt 给不同模型跑，结果不同，不应该互相替代。

**Prompt 标准化的保守策略：** 对 prompt 做哈希前会做一些清洗（统一换行符、折叠连续空白、NFC 标准化 Unicode），但**不做语义归一化**。为什么？因为"这两个 prompt 语义相同"本身就需要 LLM 判断，而缓存层不应该依赖 LLM。宁可多跑一次，也不要误命中一个"看似相同实则不同"的缓存。

### 决策五：并发为什么需要三层控制？

Dynamic Workflow 可能同时跑几十到上百个 agent。不加控制会导致资源耗尽。

**第一层：全局并发上限**（Semaphore）

限制整个系统同时在跑的 agent 数量。防止机器资源（CPU、内存、API 连接数）被耗尽。通常设为 CPU 核数的 2 倍。

**第二层：Run 级配额**

每个 workflow run 有自己的 agent 数量上限。防止一个失控的脚本（比如死循环里无限派 agent）吃掉所有全局配额，饿死其他 run。

**第三层：Agent 级取消**

每个 agent 有独立的取消令牌。用户可以停止单个 agent 而不影响其他正在跑的 agent。这在"某个 agent 明显跑偏了"的场景下很有用。

**为什么需要三层而不是一层？**

- 只要全局上限：一个失控的 run 可以占满所有槽位
- 只要有 run 上限：无法控制总资源消耗
- 只要有 agent 取消：无法预防，只能事后补救

三层各有分工：全局控制资源总量，run 级隔离故障域，agent 级提供精细控制。

### 决策六：事件为什么用广播而不是队列？

系统需要一个事件总线来传递状态信息（agent 启动、完成、失败等）。

**广播（broadcast）vs 队列（mpsc）的核心区别：**

- 队列：一条消息只有一个消费者，消费后消失
- 广播：一条消息所有订阅者都收到

事件总线的消费者不止一个：

- Headless 模式：输出到终端的 JSONL
- TUI 模式：渲染进度界面
- 持久化层：写入事件日志
- 调度器内部：追踪 agent 状态

如果用队列，一条事件只能被一个消费者拿走。用广播，所有消费者独立消费同一事件流。

广播还有一个关键特性：**消费者落后时丢弃旧事件，不阻塞生产者。** 这在 agent 高速运行时很重要——进度界面的渲染可以丢几帧，但不能拖慢 agent 的执行速度。

## 第四层：对照篇 —— Claude Code vs Maestro

两者实现的是同一个范式，但在关键岔路口做了不同选择。

### 核心相同点

| 方面 | 一致的选择 |
|------|-----------|
| 编排语言 | 可沙箱化的脚本语言 |
| 安全模型 | 脚本只能协调，agent 执行 IO |
| 质量模式 | 对抗式交叉验证 |
| 后台执行 | 运行时独立于用户会话 |
| 确定性 | 脚本持有编排逻辑 |
| 事件驱动 | 广播式状态通知 |

这些相同点不是巧合——它们是 Dynamic Workflow 范式本身的特征，不依赖于具体实现。

### 关键差异

#### 差异一：脚本语言 —— JS vs Lua

| | Claude Code | Maestro |
|---|---|---|
| 语言 | JavaScript | Lua |
| 沙箱方式 | V8 isolate | mlua + 裁剪标准库 |
| 生态优势 | npm 生态、开发者基数大 | 极度轻量、启动 < 1ms |
| 风险 | JS 的 eval/prototype pollution 难以完全封锁 | Lua 生态小、开发者不熟悉 |

**选择逻辑：** Claude Code 选 JS 是因为它的用户就是开发者，JS 是最熟悉的语言。Maestro 选 Lua 是因为它的定位更偏向"嵌入式编排引擎"——Lua 的轻量和可控更适合嵌入到更大的系统中。

这不是"谁更好"的问题，而是**目标用户不同**。

#### 差异二：恢复能力 —— 会话内 vs 跨会话

| | Claude Code | Maestro |
|---|---|---|
| 恢复范围 | 同一会话内 | 跨会话 |
| 机制 | 运行时内存缓存 | blake3 cache key + checkpoint 持久化 |
| 代价 | 简单，退出即丢失 | 需要确定性哈希、checkpoint 文件 |

Claude Code 的 resume 只在当前会话内有效。退出 CLI 后，下次启动从头跑。

Maestro 通过 checkpoint 文件实现了跨会话恢复。代价是更复杂的缓存机制——你需要确保 cache key 是确定性的，prompt 的微小变化不会导致误命中或误缺失。

**选择逻辑：** Claude Code 的假设是用户不会频繁退出（它是交互式工具）。Maestro 的假设是 workflow 可能运行很长时间，崩溃和重启是常态。

#### 差异三：后端绑定 —— 一家 vs 可插拔

| | Claude Code | Maestro |
|---|---|---|
| LLM 后端 | 仅 Claude | 任意（trait 抽象） |
| 脚本生成 | Claude 自己写 | LLM agent 动态生成（任意 backend，对齐 Claude DW） |
| 运行时 | 内嵌在 Claude Code CLI | 独立的 Rust 库 |

Claude Code 的整个链路——从脚本生成到 agent 执行——都绑定在 Anthropic 生态里。这是它的商业模式决定的。

Maestro 通过 `AgentBackend` trait 把 LLM 后端完全解耦。你可以用 Claude、用 GPT-4、用本地模型，甚至用 Mock 后端做测试。这让它更像一个**通用的编排引擎**。

**代价：** 可插拔意味着你失去了 Claude Code 那种"开箱即用"的体验——你需要自己配置后端、选择模型、管理 API key。

#### 差异四：成熟度

| | Claude Code | Maestro |
|---|---|---|
| 状态 | 研究预览，但已集成进产品 | v0.1，核心框架已实现 |
| 实际后端 | 生产级 Claude API | 仅 MockBackend 完整可用 |
| 用户界面 | CLI + Desktop + IDE + TUI 进度视图 | CLI 为主，TUI 未实现 |
| 内置工作流 | `/deep-research` 可直接使用 | 需用户自己编写脚本 |

这个差距是客观存在的。Maestro 的架构设计是完整的，但在"能直接拿来用"这件事上和 Claude Code 有显著距离。

### 分岔点总结

两个项目在"Dynamic Workflow 是什么"上完全一致，在"Dynamic Workflow 为谁服务"上分道扬镳：

- **Claude Code**：为 Claude 用户的单一产品服务 → 绑定生态、开箱即用
- **Maestro**：为需要通用 agent 编排能力的开发者服务 → 解耦后端、可嵌入

## 第五层：坦诚篇 —— 还没有解决的问题

Dynamic Workflow 作为一个范式，目前有几个根本性的挑战没有被很好地解决。

### 挑战一：脚本质量依赖 LLM 的一次性生成

Workflow 的编排脚本是 LLM 生成的。如果 LLM 生成的脚本有 bug——循环条件写错了、agent 的 prompt 不够精确、扇出的粒度不对——整个 workflow 的结果就会有问题。

而且，**脚本的正确性很难自动验证**。你不能在运行前就确定"这个脚本会产生好的结果"。你只能跑一遍看结果，然后调脚本再跑。

可能的改进方向：
- 脚本的测试框架（dry-run 模式，用缓存模拟）
- 脚本的 diff 和迭代工作流（跑一遍 → 看结果 → 调脚本 → 跑变更部分）
- 脚本的类型化接口（约束脚本只能做合理的事）

### 挑战二：成本控制的粒度不够

对抗性收敛可能让 LLM 调用量膨胀 6-10 倍。但目前的成本控制是粗粒度的——你只能在启动前决定"用不用 converge"、"最多跑几轮"。

缺少的是：
- **运行时自适应**：如果前两轮的 finding 已经很少被淘汰了，提前终止
- **质量/成本滑块**：用户说"我愿意花 $2 做这次分析"，系统自动调整 producer/adversary 数量和轮数
- **分级策略**：对低风险文件用 parallel，对高风险文件用 converge

### 挑战三：人机协作的断层

Workflow 的设计假设是"全自动"——启动后不需要人干预。但现实中，很多任务需要人在关键节点做判断：

- "迁移了 80% 的文件，剩下的 20% 结构特殊，需要人工决定怎么处理"
- "安全审计发现了 5 个高危漏洞，需要人工确认优先级"

目前的工作流要么停下来等（但"不能等待用户输入"是 workflow 的设计约束），要么硬着头皮自动决策。中间地带——"暂停、汇报、等指令、继续"——是一个尚未解决的问题。

### 挑战四：编排脚本的复用边界

Claude Code 允许保存成功的 workflow 为命令，下次直接调用。但"可复用"和"需要适应"之间存在张力：

- 同一个安全审计 workflow，在 Java 项目和 Python 项目上跑，agent 的 prompt 需要调整
- 同一个迁移 workflow，从 v1 迁到 v2 和从 v2 迁到 v3，策略可能不同

脚本的参数化（通过 args）只解决了表层问题。更深层的挑战是：**如何让一个 workflow 模板在不同的上下文中自动适应？** 这可能需要引入一个"适配层"——让 LLM 在执行保存的 workflow 前先根据当前上下文做一轮适配。

### 挑战五：评估标准缺失

"对抗性收敛比简单并行好"——这个直觉是对的，但缺乏量化证据。我们缺少：

- 收敛存活率 vs 人工标注准确率的对比
- 不同轮数、不同 producer/adversary 比例下的质量曲线
- 不同任务类型（代码审查、安全审计、调研）下的最优参数组合

没有这些数据，"用 converge 还是 parallel"仍然是一个凭经验的决定，而不是数据驱动的决定。

---

## 结语：Dynamic Workflow 的本质

回到最根本的问题：Dynamic Workflow 到底改变了什么？

它改变的是**AI agent 系统的可靠性模型**。

传统模式下，系统的可靠性 = LLM 单次调用的可靠性。LLM 有 70% 准确率，系统就是 70% 可靠。

Dynamic Workflow 通过三个机制改变了这个等式：

1. **结构化输出合约**：把 LLM 的自由文本输出约束为 schema，让结果可以被程序化验证
2. **对抗性收敛**：用多轮对抗验证淘汰不可靠结论，让存活结果的可靠率远高于单次调用
3. **确定性编排 + 缓存恢复**：让系统行为可预测、可重跑、可从故障中恢复

结果是：**即使每个 LLM 调用只有 70% 准确率，经过结构化编排后的系统输出可以达到 95%+ 的可靠率。**

这不是靠"用更好的模型"实现的——而是靠"更好的系统设计"实现的。这正是 Dynamic Workflow 作为一个工程范式的价值所在。
