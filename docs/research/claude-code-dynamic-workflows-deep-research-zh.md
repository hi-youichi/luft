# Claude Code 动态工作流

## 执行摘要

- 动态工作流将编排从 Claude 的对话式逐轮循环移入沙箱运行时中执行的脚本，解决了上下文饱和、不可重复性和缺乏质量保障的问题。
- "谁持有计划"框架区分了工作流（计划由代码持有，结果在脚本变量中）与子代理、技能和代理团队（计划由 Claude 持有，结果在上下文窗口中）。
- 脚本本身没有文件系统、shell 或网络访问权限——所有 I/O 都委托给通过 `agent()` 调用生成的子代理。
- 会话内的确定性恢复通过日志实现，该日志以任务哈希为键存储已完成的代理结果；恢复时，已完成的代理调用会跳过重新执行。
- 核心原语包括 `agent()`、`parallel()`（栅栏）、`pipeline()`（流式阶段）、`phase()`、通过 JSON Schema 的结构化输出、预算感知执行以及嵌套的 `workflow()` 调用。
- 存在三个已知限制：运行中无法人工介入、一次性 LLM 脚本生成无测试框架、粗粒度的成本控制缺乏运行时自适应机制。
- 收敛-对抗式质量模式缺乏已发布的、针对人工标注准确率的比较基准测试。

## 背景与定义

**动态工作流**是 Claude 针对给定任务即时编写的脚本，由沙箱运行时执行，并行编排数十到数百个子代理。中间结果存在于脚本变量中，而非模型的上下文窗口中；只有最终的整合答案返回对话。

区分四种机制的框架取决于**"谁持有计划"**——谁决定下一步运行什么以及中间状态存在哪里：

| 机制 | 谁决定下一步 | 中间结果存在哪里 | 规模 |
|---|---|---|---|
| **子代理** | Claude，逐轮 | Claude 的上下文窗口 | 每轮少数几个 |
| **技能** | Claude，遵循指令 | Claude 的上下文窗口 | 与子代理相同 |
| **代理团队** | 领导代理，逐轮 | 共享任务列表 | 少量长期运行的对等代理 |
| **工作流** | **脚本**（代码） | **脚本变量** | 每次运行 10–100，最多 1000 |

在子代理、技能和代理团队中，Claude 是编排者：它逐轮决定下一步生成什么，每个结果都回到其上下文窗口中。工作流将计划移入代码——脚本持有循环、分支、扇出、扇入和结果，因此 Claude 的上下文只保留最终答案。

这一范式转变解决了三个根本瓶颈：

1. **上下文饱和**——在对话模型中，每个子代理的结果在上下文窗口中不断累积直至溢出。工作流的中间状态存在于脚本变量中（几乎无上限），因此它可以协调数百个代理而不会耗尽 Claude 的上下文。

2. **不可重复的编排**——逐轮编排是短暂的；同一任务运行两次可能走上完全不同的路径。工作流脚本是确定性代码——相同输入产生相同执行路径——使其可重新运行、可恢复（会话内）、可通过 diff 审计，并可保存为 `/command` 以供复用。

3. **缺乏内置质量保障**——单次 LLM 调用没有交叉验证。工作流将对抗式收敛等质量模式编码化：多个独立生产者生成发现结果，对抗代理尝试逐一反驳，只有经过迭代投票轮次后存活的发现才被上报。

## 架构与执行模型

**脚本沙箱。** Claude Code 在 Node `vm` 沙箱中执行 JavaScript，并通过 OS 级机制进一步隔离（Linux 上 `bubblewrap`，macOS 上 `Seatbelt`）。脚本不能直接访问 `fs`、`child_process` 或网络 API。所有副作用都通过代理调用完成。

**通过日志的确定性恢复。** 恢复依赖于存储已完成代理结果的日志，以缓存键哈希为键。恢复时，运行时从检查点文件重建索引。当调用 `agent()` 时，首先查询日志——命中则跳过执行并直接返回缓存结果。此恢复是**会话范围**的（退出时缓存丢失）。`parallel()` 原语在向调度器提交任何批次之前，对每个项应用相同的缓存检查逻辑。Claude Code 的缓存机制是 API 层面基于前缀的 prompt 缓存。

## 核心原语 / 能力

官方 Claude Code 文档概念性地描述了工作流概念。以下原语源自已记录的行为和社区扩展。

**`agent(prompt, opts?)`**——核心子代理生成器。将代理的最终文本作为字符串返回，如果提供了 JSON Schema 则返回经过验证的对象。额外选项包括 `label`、`phase` 和 `model`。

**`parallel(thunks)`**——栅栏：并发生成所有 thunk，等待每一个完成后再返回。失败的 thunk 解析为 `null`（永不拒绝）。

**`pipeline(items, stages)`**——多阶段处理，具备真正的流式能力（A 项在阶段 3 时 B 项可以在阶段 1）。

**`phase(title)`**——TUI 的进度分组。后续的 `agent()` 调用被归入此标题下。

**`export const meta = { name, description, phases }`**——必需的脚本序言。必须是纯字面量（无变量、插值、展开）。

**结构化输出（JSON Schema）**——在工具调用层强制执行，schema 不匹配时重试。

**`workflow(nameOrRef, args?)`**——嵌套工作流调用。共享父级的并发上限和中止信号。

**`args`**——向已保存工作流注入参数。直接作为数组/对象传递。

**内置 `/deep-research`**——从多个角度扇出网络搜索，获取并交叉验证来源，对每个声明进行对抗式投票，返回带有引用的报告，其中被反驳的声明已被过滤（需要 WebSearch 工具）。

**触发机制。**（1）**`ultracode`**——`/effort ultracode` 将 `xhigh` 推理力度与每个实质性任务的自动工作流编排相结合。会话期间持续生效。（2）**自然语言**——在提示中包含关键词 `workflow`（或 `ultracode`）。（3）**`/workflows` 命令**——列出正在运行/已完成的运行；`s` 将运行保存为可复用命令。

**执行约束。** 最多 16 个并发代理（低核数机器上更少），每次运行总共 1000 个代理。脚本本身无文件系统/shell 访问权限。脚本为纯 JavaScript（不支持 TypeScript 语法）。通过 `resumeFromRunId` 恢复（会话范围）。

## 限制与待解决问题

**运行中无法人工输入。** 官方 Claude Code 明确记录了这一点——子代理始终在 `acceptEdits` 模式下运行，shell/网络的权限提示可能会触发，但没有工作流级别的暂停来等待人工输入。

**脚本质量取决于一次性 LLM 生成。** 编排脚本由 LLM 编写，没有测试框架。Claude Code 允许用户在启动前查看/编辑脚本。

**粗粒度成本控制。** Claude Code 提供代理上限（16 并发，1000 总计）且没有编程式预算 API。

**评估缺口。** 收敛-对抗式模式（生产者 → 对抗者 → 投票 → 存活）缺乏针对各任务类型人工标注准确率的已发布比较基准测试。默认值（投票阈值 0.7，最多 3 轮）没有已发布的准确率数据。

## 置信度与注意事项

草稿中的以下声明无法验证或已知不准确：

1. **"V8 isolate"沙箱**——草稿声称 Claude Code 使用"V8 isolate"。官方文档描述的是 Node `vm` 沙箱 + OS 级隔离（bubblewrap/Seatbelt）。"V8 isolate"这个术语未在任何官方文档中使用，具有误导性。

2. **缓存键 `blake3(backend_id + model + prompt + phase)`**——此公式来自第三方实现，而非官方 Claude Code。Claude Code 使用标准 Anthropic prompt 缓存（基于前缀），而非自定义的 blake3 任务哈希。

3. **原语 `agent()`、`parallel()`、`pipeline()`、`phase()` 作为文档化的 Claude Code API**——这些确切的函数名在社区 fork 中有文档记录。官方 Claude Code 文档未发布具有这些签名的特定 JavaScript 函数级 API；它只是概念性地描述了工作流概念。

4. **Claude Code 中 `budget` 的 `{ total, spent(), remaining() }`**——此签名仅存在于社区 fork 中。官方 Claude Code 未记录编程式预算 API。

5. **"Bun Zig→Rust 移植（~750K 行，99.8% 测试通过，11 天）"**——此案例研究引用自 Anthropic 官方博客和 Bun PR #30412。它看起来可信，但在本次审查中无法通过实时数据独立验证。该统计数据在多个来源中被广泛报道。

6. **`Date.now()`、`Math.random()` 和无参数的 `new Date()` 抛出异常**——此行为特定于社区 fork 的确定性强制措施。官方 Claude Code 文档未指定哪些全局变量受限。
